# Secrets

dagstack/config does not store secrets — it **transports** them correctly and **masks them in diagnostics**. Storage of the secrets themselves remains the responsibility of external tools (env variables, Vault, Kubernetes Secret, cloud secret managers).

## The basic rule

**A secret is an env variable that reaches the configuration through `${VAR}`.** Never write a secret value into `app-config.yaml` or `app-config.${ENV}.yaml`:

```yaml title="WRONG — the secret ends up in git"
database:
  password: "s3cr3t-prod-pw"
```

```yaml title="RIGHT — the secret comes from the environment, never committed"
database:
  password: "${DB_PASSWORD}"
```

## Auto-masking by field name

When the configuration is rendered into logs / diagnostics / error messages, secret fields are automatically replaced with `[MASKED]`. The pattern list is fixed normatively in `config-spec/_meta/secret_patterns.yaml`:

| Exact match | Suffix |
|---|---|
| `api_key` | `*_secret` |
| `secret_key` | `*_token` |
| `access_token` | `*_password` |
| `password` | `*_key` |
| `client_secret` | |

Examples of fields that get masked:

- `database.password` (exact)
- `cache.auth_token` (suffix `_token`)
- `auth.jwt_secret` (suffix `_secret`)
- `payment.stripe_api_key` (suffix `_key`)
- `webhook.signing_password` (suffix `_password`)

Not masked:

- `database.host` — does not match any pattern.
- `database.url` — URLs may contain credentials (`postgresql://user:pass@host`), but the pattern is not designed for that; pass `password` as a separate field.
- A custom field `internal_key_id` — contains `_key`, BUT: matching is by **suffix** `_key` → **it matches**. If this is not a secret (just an id), rename it: `internal_id` / `key_name`.

## Diagnostics are masked

Masking is applied automatically inside the `ConfigError.details` message: when the loader or a typed getter raises an error on a value in a secret-named field, the exception text shows `[MASKED]` instead of the raw value. You do not have to opt in.

```text
ConfigError(
  reason=type_mismatch,
  path=database.password,
  details=expected string, got int with value [MASKED]
)
```

For your own diagnostic output the bindings expose three primitives:

Use them when you build a custom dump of the configuration tree, when you log specific fields, or when you want a stable check for whether a field name is a secret.

## `.local.yaml` — for non-committed overrides

If you need to temporarily override a secret locally for debugging, use `app-config.local.yaml`:

```yaml title="app-config.local.yaml"
database:
  password: "local-dev-pw"    # ok — this file is in .gitignore
```

The file **must** be listed in `.gitignore`. The standard `.gitignore` template for dagstack projects contains:

```
# dagstack config — local developer overrides, never committed.
app-config.local.yaml
```

An alternative is a `.env` file with a variable that is referenced through `${VAR}`:

```dotenv title=".env (gitignored)"
DB_PASSWORD=local-dev-pw
```

## Public secret managers (Vault / K8s Secret)

In production, secrets typically come from a centralized secret manager:

**HashiCorp Vault** through envconsul / agent:

```bash
# envconsul launches the application with env vars sourced from Vault
envconsul -config=vault.hcl -- python main.py
```

**Kubernetes Secret** through env:

```yaml title="kubernetes/deployment.yaml"
env:
  - name: DB_PASSWORD
    valueFrom:
      secretKeyRef:
        name: db-credentials
        key: password
  - name: PAYMENT_API_KEY
    valueFrom:
      secretKeyRef:
        name: payment-secrets
        key: stripe-api-key
```

dagstack/config sees `${DB_PASSWORD}` in YAML — the source of the variable (env / Vault / K8s) is transparent to it.

In Phase 2+, direct adapters (`VaultSource`, `KubernetesSecretSource`) will arrive — a component will be able to read a secret from Vault without an env intermediary. But env is the universal "lowest common denominator" for the current release.

## What to do if a secret leaks

If you discover that a secret reached git (for example, `app-config.yaml` with a plaintext password or an API key):

1. **Revoke the secret immediately** in the relevant service (in a database, change the user's password; with an API provider, revoke the key).
2. Issue a new secret and update the env variables in CI / production.
3. Rewrite the YAML to use `${VAR}` substitution.
4. Rewrite the git history (git filter-branch / BFG Repo Cleaner) — if the repository is not yet public.
5. If the repository was public, treat the secret as compromised regardless of history rewrites.

## Adding your own secret patterns

The pattern list is **fixed** in v0.1 / v0.2 — it mirrors `config-spec/_meta/secret_patterns.yaml` and the bindings do not yet expose a way to extend it at load time. Runtime customization (`Config.load(secret_patterns=...)` or equivalent) is on the Phase 2+ roadmap.

If you need to mask a field whose name does not match the standard list, do it in your own diagnostic dump using `mask_value` / `maskValue` / `MaskValue`:

<Tabs groupId="lang" queryString="lang">
  <TabItem value="python" label="Python">
    ```python
    # Custom dump that also masks the project-specific 'connection_string'.
    def custom_masked_string(name: str, value: object) -> object:
        if name == "connection_string" or is_secret_field(name):
            return MASKED_PLACEHOLDER
        return value
    ```
  </TabItem>
  <TabItem value="typescript" label="TypeScript">
    ```ts
    // Custom dump that also masks the project-specific 'connection_string'.
    function customMaskedString(name: string, value: unknown): unknown {
      if (name === "connection_string" || isSecretField(name)) {
        return MASKED_PLACEHOLDER;
      }
      return value;
    }
    ```
  </TabItem>
  <TabItem value="go" label="Go">
    ```go
    // Custom dump that also masks the project-specific "connection_string".
    func customMaskedString(name string, value any) any {
        if name == "connection_string" || IsSecretField(name) {
            return MaskedPlaceholder
        }
        return value
    }
    ```
  </TabItem>
</Tabs>

The 9 standard patterns already cover roughly 95% of cases; rename project-specific secret fields to match a standard suffix (`*_secret`, `*_token`, `*_password`, `*_key`) when you can — that keeps masking automatic.
